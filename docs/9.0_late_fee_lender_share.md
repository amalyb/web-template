## TL;DR

Today every dollar of the daily late fee goes to Sherbrt. The lender ŌĆö
whose item is the one tied up ŌĆö gets nothing. This PR adds an optional
provider-side line item to each daily late-fee charge so the lender
receives 50% of the fee (default; configurable 0ŌĆō100%) at the next
payout transition (`complete-return` or `complete-replacement`).

Borrower experience is unchanged: same $15/day, same $75 cap, same SMS
copy (steps 9.1ŌĆō9.6 in `sherbrt_transaction_comms_v15.xlsx`). The split
happens entirely on the back end via line items; the borrower charge
amount is identical either way.

The replacement-value charge (manual, operator-fired) is NOT affected by
this flag. Replacement remains 100% lender (operator builds line items
manually at replacement time ŌĆö it's reimbursement for a lost asset, not
a penalty).

---

## Why 50%

Rationale lives in `CLAUDE_CONTEXT.md` (May 29 2026 entry). Short
version:

- **The harm is the lender's.** It's their asset being held past return.
  Routing half the fee to them maps compensation onto the party that
  actually bore the cost.
- **Lender retention.** Sherbrt is supply-constrained. A lender who gets
  nothing for a late return is a lender who eventually delists. At 50%
  the lender earns up to $37.50 over the full 5-day cap ŌĆö meaningful
  against a typical ~$21 lender payout.
- **Sherbrt still nets positive at 50%.** $7.50/day comfortably covers
  Stripe processing (~$0.74 on a $15 off-session PI), SMS, day-6
  operator/CS time, and elevated dispute risk on late-fee charges.
- **Deterrent unchanged.** Borrower still owes $15/day; the split is
  invisible to them.
- **Industry precedent.** P2P rental + car-share platforms generally
  pass late/overage fees substantially to the asset owner.

The percentage is exposed as `LENDER_LATE_FEE_SHARE_PCT_OVERRIDE` so the
exact share can be tuned in staging or in prod (env-flip, no redeploy)
without a code change.

## Implementation

### Code changes ŌĆö `server/lib/lateFees.js`

1. New constants at top of file:
```js
const LATE_FEE_LENDER_SHARE_ENABLED =
  String(process.env.LATE_FEE_LENDER_SHARE_ENABLED || 'false').toLowerCase() === 'true';

const LENDER_LATE_FEE_SHARE_PCT = /* parse + clamp 0..100, default 50 */;
```

2. New pure helper `buildLateFeeLineItems({ lateFeeCents, lenderSharePct })`
   that returns either `[customerLine]` (split off / pctŌēż0) or
   `[customerLine, providerLine]`. Provider line follows the
   `line-item/provider-commission` shape: percentage-based, no
   quantity, `includeFor: ['provider']`.

3. New pure helper `lineItemEffectiveCents(lineItem)` ŌĆö computes
   `unitPrice * percentage / 100` (percentage-based) or
   `unitPrice * quantity` (quantity-based). Used for `amounts`,
   `wouldCharge`, and `chargeHistory.items` so all summaries report
   effective money moved, not unitPrice.

4. `applyCharges()` now builds line items via the helper, with
   `effectiveLenderSharePct = LATE_FEE_LENDER_SHARE_ENABLED ? LENDER_LATE_FEE_SHARE_PCT : 0`.
   Cap filter still matches `code === 'late-fee'` only ŌĆö the new
   `late-fee-lender-share` code never inflates the count-based cap.

5. Return value adds `lenderShareCents`, `platformShareCents`,
   `lenderShareEnabled` for operator visibility (digest emails,
   structured logs). Existing fields (`charged`, `items`, `amounts`,
   `reason`, `lateDays`, `scenario`, `wouldCharge`) are preserved.

### process.edn ŌĆö NO change required

`transition/privileged-apply-late-fees-non-return` already runs

### Payout mechanics ŌĆö ASSUMPTION, must verify empirically before PR-4

ŌÜĀ’ĖÅ This is the load-bearing premise of the whole change. It is NOT
verified by anything in this repo and rests on Sharetribe server
semantics not observable in JS. Code review (May 29 2026) correctly
flagged this ŌĆö see the verification gate in PR-2 below.

The assumption: `stripe-create-payout` only fires on
`transition/complete-return` or `transition/complete-replacement`
(process.edn:87ŌĆō96). The lender's late-fee share, added as a
provider-included line item by `privileged-apply-late-fees-non-return`
(a mid-flow self-loop, `:from :state/accepted :to :state/accepted`,
process.edn:148ŌĆō163), accumulates into the cumulative provider payout
total and is transferred whenever one of those payout transitions
runs:
- Late-but-eventually-returned tx ŌåÆ `complete-return` fires on the
  return delivered scan (already wired in
  `server/webhooks/shippoTracking.js`).
- Never-returned tx ŌåÆ `complete-replacement` fires manually when
  operator processes the replacement (already wired in
  `server/scripts/sendOverdueReminders.js` for the replacement path).
- A tx stuck in `:state/accepted` forever (no return scan, no operator
  replacement) would strand the lender share in the platform balance ŌĆö
  acceptable: that's already an investigation case at day 6+ and the
  operator can decide to fire complete-replacement manually.

Why this is genuinely uncertain: `privileged-set-line-items` is
documented as REPLACING `tx.attributes.lineItems` with a new array.
That suggests the late-fee transition wipes the rental + commission
line items and replaces them with `[late-fee]` (customer-only). If
`stripe-create-payout` then read live line items at payout time, the
lender's normal rental payout would already be destroyed today (no
provider line items left ŌåÆ payout total = 0). Since that evidently
isn't happening in prod (lenders still get paid after late-fee txs
complete), Sharetribe must be doing something to preserve/accumulate
payout across transitions ŌĆö but exactly what is not visible from this
repo. The pessimistic failure modes are:
- Payout snapshots at booking-capture time and is immutable
  afterward ŌåÆ lender share NEVER transfers, stranded in platform.
- Payout reads only the LAST `set-line-items` array ŌåÆ last day's
  `[late-fee, late-fee-lender-share]` becomes the entire payout basis
  ŌåÆ lender gets only $7.50 instead of full rental + $7.50.

Neither is what we want. The right empirical check is below.

## Edge cases

### Borrower disputes / chargebacks
- Borrower disputes the late-fee charge with their issuing bank.
- Dispute hits Sherbrt's Stripe account on the customer PI. The
  lender's already-accrued share is NOT automatically clawed back
  by Stripe (it's separate Connect bookkeeping).
- Operator handling: when investigating a chargeback, decide whether
  to also reverse the lender Transfer manually. Same shape of
  decision they already make for refunds on accepted bookings.

### Tx never reaches a payout transition
- Tx sits in `:state/accepted` indefinitely (no return scan, no
  operator replacement).
- Lender share accrues in line items but never transfers to lender.
- Acceptable: any tx that goes 7+ days past due is already a day-6
  investigation case; operator decides whether to fire
  `complete-replacement` (lender gets full replacement value + the
  accrued late-fee share) or `cancel` (no payouts ŌĆö but this is a
  policy decision per case).

### Rounding at odd percentages (review S1)
- 33% of $15 = $4.95 (Math.round half-up). Sherbrt's share = $10.05.
  `lenderShareCents + platformShareCents === LATE_FEE_CENTS` is
  guaranteed BY CONSTRUCTION (platform = LATE_FEE_CENTS - lender), so
  the summary never leaks pennies internally.
- At 50% / 100% / 25% (the realistic configs), no rounding occurs.
- **Caveat:** `lenderShareCents/platformShareCents` (in the digest +
  chargeHistory) are JS-side ESTIMATES computed via Math.round.
  Sharetribe's own line-total rounding may disagree by 1 cent at
  non-divisor percentages (33, 67, etc.). The authoritative value for
  any audit is `tx.attributes.lineItems[].lineTotal` after the
  transition lands. At common configs the two always agree exactly.

### Stripe Connect minimum payout
- Stripe Connect has a $1 minimum for some account types. A single
  $7.50 daily share is above that, but the system accumulates
  shares across all of a lender's transactions before any individual
  payout, so this is not a practical issue at default values.

### Process-version compatibility
- Already handled by `MIN_PROCESS_VERSION_FOR_LATE_FEES = 4`. Old
  txs skip cleanly with `processVersion-too-old`. No additional
  gate needed for the split.
