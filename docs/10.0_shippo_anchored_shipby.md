## TL;DR

Today's ship-by date is computed from a static `SHIP_LEAD_DAYS` env var (currently `=1` on the web service, unset on the Lender Shipping Reminders cron, causing silent desync — see the April 23, 2026 env-drift incident documented in `CLAUDE_CONTEXT.md`). The web service and cron can disagree on what day counts as "ship-by day" for the same transaction, which breaks the 3.1b end-of-day reminder.

This PR replaces the static model with a Shippo-anchored model: ship-by is derived from the actual carrier's `estimated_days` for the selected rate, persisted at label purchase, and read from `protectedData.outbound.shipByDate` by all downstream consumers. It also bundles related hardening:

- Expand preferred services to include expedited options so short-lead bookings can actually ship on time **AND** fix the rate-selection functions (both `estimateOneWay` at checkout and `pickCheapestAllowedRate` at accept) that today don't actually respect the `preferredServices` config list
- Split outbound vs return rate-selection logic (outbound deadline-aware, return always-cheapest)
- Rate-lock at borrower checkout (save `rateObjectId`) and **always purchase that exact rate at accept** — no feasibility re-check, no fallback re-selection, no delta to reconcile (CC review "Option 6", April 23, 2026)
- Tighten `transition/expire` from 6 days to 24 hours so the checkout-to-accept gap stays narrow
- Add 22h final-warning escalation SMS (bypasses quiet-hours) to the lender request reminder cron so lenders don't silently miss the 24h window
- Add a 12-hour scan-lag grace buffer to auto-cancel so late-night carrier scans don't cause premature cancels
- Fix the `processVersion !== 3` gate in `sendAutoCancelUnshipped.js` so auto-cancel keeps working after the v5 alias flip

## Current State — verified by code review (April 23, 2026)

**Ship-by date computation (`server/lib/shipping.js:24, 149-211`):** `computeShipByDate` in the core shared lib always recomputes from `bookingStart − SHIP_LEAD_DAYS` and does not read persisted values. `sendShippingReminders.js:133-158` has a local `getShipByDate()` helper that DOES read `protectedData.outbound.shipByDate` first and falls back to `computeShipByDate` — but when the persisted value is missing or inaccessible in the Integration-SDK tx fetch, it fell through to the fallback. The real fix is to move the persisted-value preference INTO the shared `computeShipByDate` so every downstream consumer benefits automatically.

**Env drift, confirmed April 23, 2026:** Web service has `SHIP_LEAD_DAYS=1`. Lender Shipping Reminders cron has the var unset (defaults to `2`). Web service sent 3-SMS saying "Ship by Apr 22" (Apr 23 − 1). Cron fell through to the `computeShipByDate` fallback (persisted value absent or not surfaced in the cron's tx fetch shape) and recomputed with its own `SHIP_LEAD_DAYS=2` → ship-by = Apr 21. `isShipByDay` compared Apr 21 against today Apr 22 → false → skipped 3.1b for the tx. Cron summary `EndOfDay=0, Processed=65` is the smoking gun — every `[ship-by:static] { chosenLeadDays: 2 }` line confirms lead=2 in the cron.

**Preferred services (`server/config/shipping.js:16-20`):** Only 3 services enabled — USPS Priority Mail, USPS Ground Advantage, UPS Ground. No expedited options. More importantly, per CC review (April 23, 2026): **neither of the rate-selection functions actually respects this config list correctly today.**
- `pickCheapestAllowedRate` (transition-privileged.js:188-228) filters by `provider` (UPS/USPS only) and `servicelevel.token` (for the ups_ground preference). It never reads `preferredServices`.
- `estimateOneWay` (shipping.js:491-506) DOES read `preferredServices` but its name-builder uses `(r.carrier || r.provider) + ' ' + (r.service || r.provider_service)`. Shippo's modern SDK returns `{provider, servicelevel: {name, token}}` — so `r.service` is usually undefined/an object, the filter matches nothing, and the fallback `filtered.length ? filtered : allRates` silently selects cheapest-of-all-rates rather than cheapest-preferred.

So for short-lead cross-country bookings today, `pickCheapestAllowedRate` hits the "last resort: absolute cheapest" branch (`transition-privileged.js:227`) and picks a too-slow rate that won't arrive by `bookingStart`, because the preferred-services concept isn't actually wired up. Expanding the config list alone is a no-op until we also fix both functions.

**Checkout vs accept rate-selection mismatch:** `estimateOneWay` (at checkout, `shipping.js:503`) sorts by amount, picks cheapest preferred — NO deadline filter. `pickCheapestAllowedRate` (at accept, `transition-privileged.js:188`) applies deadline filter. For short-lead bookings, these can select different rates at different prices. The delta is silently absorbed via Sherbrt's Shippo account (not charged to borrower, not logged).

**Return shipping (`transition-privileged.js:1123-1125`):** Uses `pickCheapestAllowedRate(returnRates, { shipByDate, preferredProviders })` with OUTBOUND's `shipByDate` as the deadline. Since outbound's shipByDate is in the past by the time a return ships, the function always hits the "last resort: absolute cheapest" branch. Works accidentally, but the coupling is wrong.

**Lender acceptance expire window (`ext/transaction-processes/default-booking/process.edn:72-86`):** Currently `:fn/min` of `preauthorized + 6 days`, `bookingStart + 1 day`, `bookingEnd`. For long-lead bookings, lenders effectively have 6 days to accept. This is too wide — enables the checkout-to-accept rate-selection drift.

**Lender request reminder (`server/scripts/sendLenderRequestReminders.js:113,117`):** One-shot 60-minute nudge only. `MAX_AGE_MS = 13h`. No escalation between 60 min and 13h. If a lender misses the 60-min nudge and isn't online, they can silently blow past a tightened 24h expire.

**Auto-cancel scan-lag gap (`sendAutoCancelUnshipped.js:186-201`):** No grace period on the cancel deadline. Lender drops package at 11:50pm lender-local on booking-start date D, carrier scans at 6am D+1. If cron tick runs at 12:00am D+1 (within the gap), `hasOutboundScan()` returns false, `isPastCancelDeadline()` returns true, auto-cancel fires prematurely. Overdue cron has `daysLate <= 1 → scan-lag-grace` at `lateFees.js:294-296`; auto-cancel has no equivalent.

## Policy Decisions Locked In

The following were agreed during scope review (April 23, 2026). Do NOT re-ask in PR implementation.

1. **Ship-by anchored to Shippo's `estimated_days`**, not static env var, not distance-bucketed.
2. **Preferred services expanded to 6**: USPS Priority Mail, USPS Ground Advantage, USPS Priority Mail Express, UPS Ground, UPS 2nd Day Air, UPS Next Day Air Saver.
3. **Outbound rate-selection:** deadline-aware, cheapest preferred service that meets `daysUntilBookingStart` with `buffer=1`. Fallback order: prefer UPS Ground if feasible → cheapest feasible preferred → absolute cheapest (last resort).
4. **Return rate-selection:** always cheapest preferred service. No deadline filter. No dependency on outbound shipByDate.
5. **CC Option 6 — Always use the locked rate at accept.** Buffer=1 at checkout only (for rate selection during checkout). At accept, there is NO feasibility re-check and NO fallback re-selection. Purchase the rate whose `object_id` was saved at checkout, full stop. Borrower preauth cost = actual label cost, always. Tradeoff: in the narrow edge case where the borrower booked right at the feasibility edge AND the lender accepts near the 24h mark AND carrier transit experiences variability, the package might arrive ~1 day late ~15% of the time. Accepted in exchange for structural simplicity and zero-delta cost consistency. (Supersedes earlier draft which had buffer=1 at both phases with a feasibility check; CC review April 23, 2026 pointed out that committing fully to locked-rate-always eliminates the entire delta-reconciliation discussion.)
6. **Acceptance expire window: 24 hours** (tightened from 6 days). Process.edn v5 bump + alias flip.
7. **Rate-lock at checkout:** persist `rateObjectId + estimatedDays + amountCents` to preauth data. At accept, ALWAYS purchase the locked rate by ID. No feasibility check, no fallback.
8. **Business-day subtraction for shipByDate:** skip Sundays and USPS holidays, KEEP Saturdays. Uses **Pacific Time** for day-of-week and holiday lookups (aligned with existing `isNonChargeableDate` at `businessDays.js:104-109`). Existing `businessDays.js` logic extended with a new `subtractBusinessDays` helper.
9. **Auto-cancel scan-lag grace: 12-hour buffer** (Option 2 from review). Log event: `scan-lag-grace`, structurally parallel to overdue's `daysLate <= 1` guard. Compounds intentionally with the existing Monday-start grace: Monday-start bookings get a 60-hour post-bookingStart cancel window (23:59 lender-local on D+1 via Monday-grace + 12h scan-lag on top). Non-Monday bookings get 36h (23:59 on D + 12h scan-lag). This is intentional — the two graces address different problems (no-carrier-Sunday vs scan-delay), so both apply.
10. **Lender request reminder escalation: add 22h final-warning reminder** (60m existing nudge stays; 12h intermediate phase NOT added — 2-phase escalation is sufficient per operator decision). Distinct Redis dedupe keys per phase. **22h final-warning bypasses quiet-hours** — a brief after-hours text when money is 2h from expiring is preferable to a silent miss. 60m phase continues to respect quiet-hours via existing `withinSendWindow` gate.
11. **Same-day accept/ship allowed** — no minimum-lead-days floor.
12. **No listing-search validation** to pre-filter infeasible bookings. If a booking becomes infeasible at accept, the "last resort: absolute cheapest" branch runs and the lender may get a too-slow rate; accept anyway. Future work.
13. **No delta-reconciliation needed at accept.** Per CC Option 6 (decision #5): always use the locked rate_id. There is no delta because checkout and accept purchase the same exact rate. The previously-deferred "checkout-vs-accept unification" follow-up is RESOLVED by this design, not deferred. No follow-up PR needed.
14. **`SHIP_LEAD_DAYS` env var stays as a fallback floor** when Shippo rate data is unavailable (e.g., Shippo outage). Not removed.
15. **Auto-cancel version gate fix:** `sendAutoCancelUnshipped.js:143` currently uses `processVersion !== 3` (hard equality). After v5 alias flip, new transactions have `processVersion === 5` and auto-cancel silently skips every one. Change to `processVersion < 3` (i.e., skip only old versions) or explicit `processVersion >= 3` — either works. Bundled into PR-4.

```js
const { preferredServices } = require('../config/shipping');
const SAFETY_BUFFER_DAYS = Number(process.env.SHIP_SAFETY_BUFFER || 1);

function pickCheapestAllowedRate(availableRates, { daysUntilBookingStart, preferredProviders = ['UPS','USPS'] }) {
  if (!Array.isArray(availableRates) || availableRates.length === 0) return null;

  const daysUntil = daysUntilBookingStart ?? 999;
  const buffer = SAFETY_BUFFER_DAYS;

  const nameOf = r => `${r.provider || r.carrier || ''} ${r.servicelevel?.name || r.service?.name || r.provider_service || ''}`.trim();

  const norm = availableRates.map(r => ({
    provider: String(r.provider || '').toUpperCase(),
    token: r.servicelevel?.token || r.service?.token || '',
    name: nameOf(r),
    amount: Number(r.amount ?? r.amount_local ?? r.rate ?? 1e9),
    estDays: Number(r.estimated_days ?? r.duration_terms ?? 999),
    raw: r,
  }));

  // Filter to preferred services FIRST (this is what was missing)
  const preferredFiltered = preferredServices.length
    ? norm.filter(n => preferredServices.includes(n.name))
    : norm;

  // Then apply provider preference order within the preferred set
  let candidates = [];
  for (const p of preferredProviders.map(p => p.toUpperCase())) {
    const subset = preferredFiltered.filter(n => n.provider === p);
    if (subset.length) { candidates = subset; break; }
  }
  if (!candidates.length) candidates = preferredFiltered.length ? preferredFiltered : norm;

  // Prefer UPS Ground if it meets the deadline
  const ground = candidates.filter(n => n.token === 'ups_ground').sort((a, b) => a.amount - b.amount);
  if (ground.length && (ground[0].estDays + buffer) <= daysUntil) return ground[0].raw;

  // Otherwise cheapest that meets the deadline
  const feasible = candidates.filter(n => (n.estDays + buffer) <= daysUntil).sort((a, b) => a.amount - b.amount);
  if (feasible.length) return feasible[0].raw;

  // Last resort: cheapest within preferred set (expedited will land here for very tight bookings)
  return candidates.sort((a, b) => a.amount - b.amount)[0].raw;
}
```

### PR-2 — Shippo-anchored shipByDate + rate-lock at checkout

Scope: `server/lib/shipping.js`, `server/api/transition-privileged.js`, `server/api-util/lineItems.js`, `server/api-util/integrationSdk.js`, `server/lib/businessDays.js`.

1. **Extend `businessDays.js`** with a new helper `subtractBusinessDays(fromDate, n, opts)`. **Use Pacific Time for day-of-week and holiday lookups**, consistent with existing `isNonChargeableDate` at `businessDays.js:104-109`:
```js
const dayjs = require('dayjs');
// TZ, USPS_HOLIDAYS, NON_CHARGEABLE_WEEKDAYS already defined above in this file

/**
 * Subtract n business days from fromDate.
 * "Business day" = any day NOT Sunday AND NOT a USPS holiday (evaluated in PT).
 * Saturday IS a business day by default (per 10.0 scope decision #8).
 *
 * @param {Date|dayjs} fromDate - input date (any TZ; will be evaluated in PT)
 * @param {number} n - number of business days to subtract
 * @param {Object} opts - { skipSaturday: boolean } (default false)
 * @returns {dayjs} PT dayjs at start-of-day, n business days earlier
 */
function subtractBusinessDays(fromDate, n, opts = {}) {
  const { skipSaturday = false } = opts;
  let d = dayjs(fromDate).tz(TZ).startOf('day');
  let remaining = n;
  while (remaining > 0) {
    d = d.subtract(1, 'day');
    const dayOfWeek = d.day(); // PT-based: 0 = Sun, 6 = Sat
    const ymd = d.format('YYYY-MM-DD'); // PT-local YMD
    if (dayOfWeek === 0) continue; // Sunday
    if (skipSaturday && dayOfWeek === 6) continue;
    if (USPS_HOLIDAYS.has(ymd)) continue;
    remaining--;
  }
  return d;
}
```
Export alongside existing helpers. `USPS_HOLIDAYS` set and `TZ` constant are already in the file. Using PT matches existing holiday-observance semantics and avoids UTC-midnight off-by-one errors around holidays.

When callers convert the result back to a UTC Date for persistence, use `.toDate()` or `.utc().toDate()` — be explicit about which side of the conversion you want. For our use case (persisting to `protectedData.outbound.shipByDate` as ISO string), the PT start-of-day converted to UTC ISO is correct: the stored value represents "PT-local midnight at the start of ship-by day."

2. **Rewrite `computeShipByDate` in `server/lib/shipping.js`** to prefer persisted value, accept an optional `transitDays` arg, and use business-day subtraction. **Important type handling:** `subtractBusinessDays` returns a `dayjs` object in PT (per step 1 above). The existing `adjustIfSundayUTC` helper at `shipping.js:130-139` calls `.getUTCDay()`, which is a native Date method — dayjs objects don't have it. Must convert via `.toDate()` before passing to `adjustIfSundayUTC`:
```js
const { subtractBusinessDays } = require('./businessDays');
const SAFETY_BUFFER_DAYS = Number(process.env.SHIP_SAFETY_BUFFER || 1);

async function computeShipByDate(tx, opts = {}) {
  // 1. Prefer persisted value (written at label purchase time)
  const persisted = tx?.attributes?.protectedData?.outbound?.shipByDate;
  if (persisted) {
    const d = new Date(persisted);
    if (!Number.isNaN(+d)) {
      console.log('[ship-by:persisted]', {
        shipByISO: persisted,
        txId: tx?.id?.uuid,
      });
      return d;
    }
  }

  // 2. Compute from booking start + transit days (if provided) or fallback
  const startISO = getBookingStartISO(tx);
  if (!startISO) return null;
  const start = new Date(startISO);
  if (Number.isNaN(+start)) return null;
  start.setUTCHours(0, 0, 0, 0);

  // Caller-provided transitDays (from selected Shippo rate) is the authoritative
  // path. Without it, fall back to static LEAD_FLOOR for safety (Shippo outage,
  // manual invocations, etc.).
  const transitDays = opts.transitDays;
  const leadDays = transitDays != null
    ? transitDays + SAFETY_BUFFER_DAYS
    : LEAD_FLOOR;

  // subtractBusinessDays returns a dayjs object in PT; convert to native Date
  // before passing to adjustIfSundayUTC (which uses Date.getUTCDay()).
  const shipByDayjs = subtractBusinessDays(start, leadDays);
  const shipBy = shipByDayjs.toDate();

  // Existing Sunday → Saturday adjust preserved for parity (belt + suspenders
  // with business-day subtraction above; only fires if subtraction somehow
  // lands on a Sunday, which shouldn't happen with the new logic).
  const ADJUST_SUNDAY = String(process.env.SHIP_ADJUST_SUNDAY || '1') === '1';
  const adjusted = ADJUST_SUNDAY ? adjustIfSundayUTC(shipBy) : shipBy;

  console.log('[ship-by:computed]', {
    startISO,
    transitDays: transitDays ?? null,
    leadDays,
    mode: transitDays != null ? 'shippo-anchored' : 'static-fallback',
    shipByISO: adjusted.toISOString(),
  });

  return adjusted;
}
```

```js
// After main transaction loop, BEFORE final summary logging:
const WATCHDOG_LOOKBACK_MS = 30 * 60 * 1000; // 30 min (covers 2 cron ticks at 15 min cadence)
const expireWindow = new Date(Date.now() - WATCHDOG_LOOKBACK_MS);

try {
  const expiredRecent = await sdk.transactions.query({
    lastTransitions: 'transition/expire',
    // NOTE: Sharetribe does not support a lastTransitionedAt server-side filter,
    // so we over-fetch and client-filter by timestamp below.
    per_page: 50,
  });

  let missedFinalCount = 0;
  for (const tx of expiredRecent.data.data || []) {
    const lastAt = tx.attributes?.lastTransitionedAt;
    if (!lastAt || new Date(lastAt) < expireWindow) continue; // outside 30-min window
    const txId = tx.id?.uuid || tx.id;
    const twentyTwoSent = await redis.get(redisKey(txId, '22h:sent'));
    if (!twentyTwoSent) {
      // Dedupe: a missed tx will appear in the 30-min lookback for 2 cron ticks
      // (at 15 min and 30 min post-expire). Without dedupe, each tick double-logs
      // the same miss, inflating the apparent miss rate by 2x for ops.
      const alreadyLogged = await redis.get(redisKey(txId, 'missedFinal:logged'));
      if (alreadyLogged) continue;
      console.log(`[lender-request-reminder][MISSED_FINAL] tx=${txId} — expired without 22h warning fired`);
      await redis.set(redisKey(txId, 'missedFinal:logged'), '1', 'EX', 60 * 60); // 1h TTL
      missedFinalCount++;
    }
  }
  if (missedFinalCount > 0) {
    console.log(`[lender-request-reminder][MISSED_FINAL_SUMMARY] count=${missedFinalCount} window=${WATCHDOG_LOOKBACK_MS}ms`);
  }
} catch (watchdogErr) {
  // Watchdog failures must not block the main cron. Log and continue.
  console.warn(`[lender-request-reminder][WATCHDOG_ERROR]`, watchdogErr?.message || watchdogErr);
}
```

- **Business-day subtraction verification matrix** (PT-based, skip Sun + USPS holidays, keep Sat):
  - `estimated_days=1` from Monday bookingStart: subtract 1 business day → Sun (skip) → Sat (keep, count=1). shipBy = Saturday.
  - `estimated_days=1` from Tuesday bookingStart: subtract 1 → Mon (keep, count=1). shipBy = Monday.
  - `estimated_days=2` from Tuesday bookingStart: subtract 1 → Mon, subtract 2 → Sun (skip) → Sat (keep, count=2). shipBy = Saturday.
  - `estimated_days=3` from Mon 2026-11-30 (week of Thanksgiving, US): subtract 1 → Sun (skip) → Sat 11/28 (count=1), subtract 2 → Fri 11/27 (keep, count=2), subtract 3 → Thu 11/26 (holiday, skip) → Wed 11/25 (count=3). shipBy = Wed 11/25.
  - `estimated_days=3` across year boundary from Mon 2027-01-04: subtract 1 → Sun (skip) → Sat 01/02 (count=1), subtract 2 → Fri 01/01 (holiday, skip) → Thu 2026-12-31 (count=2), subtract 3 → Wed 12/30 (count=3). shipBy = Wed 2026-12-30.

- **Locked-rate expiration:** Shippo rate `object_id` valid for 7 days. With 24h expire, borrower checkout to lender accept is ≤24h, well inside. If expire window is ever widened back beyond 7 days, add a rate-age check before attempting purchase (Shippo will reject a stale rate_id with a specific error — handle gracefully by re-selecting).
- **Pre-PR-2 transactions at accept time:** Transactions that went through checkout before PR-2 shipped (no `lockedRate` persisted) will hit the `NO_LOCK_FALLBACK` branch at accept and use fresh rate selection. This is a one-time migration tail. After all pre-PR-2 transactions complete, the fallback should never fire. Monitor the log line for a few weeks post-deploy; if it keeps firing on new transactions, there's a bug in the checkout-time persistence.
