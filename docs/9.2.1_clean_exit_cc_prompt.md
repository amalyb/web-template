# 9.2.1 — Clean exit for overdue cron (fixes "Timed out" false-failure)

## Context

The "Overdue / Late-Fee Reminders & Charges" cron on Render (runs daily
at 17:00 UTC, command: `npm run worker:overdue-reminders`) reports
"Failed" on every run despite completing its work correctly. Render's
event feed shows every run as "Timed out while running your code." The
stdout logs show the work completes (candidates processed, summary
printed, SMS fires when applicable) — but the process never exits.
ioredis auto-reconnects after the summary block (visible in logs as
repeated `[redis] connect / [redis] ready` lines) until Render's cron
timeout force-kills the process.

Discovered during post-9.2 validation on April 21, 2026. The other cron
scripts (sendAutoCancelUnshipped, sendShippingReminders,
sendLenderRequestReminders) exit cleanly and show "Successful run" —
this is the only one that doesn't.

Context doc pattern reference: `sendAutoCancelUnshipped.js` uses an
explicit `--once` branch with `.then(process.exit(0))`. The overdue
script's no-flag entry-point branch does NOT follow that pattern; it
calls the async function without awaiting and without an explicit exit,
leaving Redis + SDK keep-alives to hold the event loop open.

## Fix

**File:** `server/scripts/sendOverdueReminders.js` (around lines 936-938,
the no-flag else branch at the bottom of the `require.main === module`
block).

**Current:**

```js
} else {
    sendOverdueReminders();
}
```

**Replace with:**

```js
} else {
    sendOverdueReminders()
      .then(() => {
        console.log('✅ Overdue reminders script complete, exiting.');
        process.exit(0);
      })
      .catch(err => {
        console.error('❌ Overdue reminders script failed:', err);
        process.exit(1);
      });
}
```

This mirrors the entry-point exit pattern used elsewhere in the codebase
for once-per-cron scripts.

## Out of scope (explicit)

- Do NOT touch the `--daemon` branch. It has its own setInterval loop
  and intentionally doesn't exit.
- Do NOT touch the `--test` branch.
- Do NOT refactor the `sendOverdueReminders()` function body itself.
- Do NOT scope-check other scripts for the same pattern — the other
  crons already exit cleanly (verified by their "Successful run" status
  in Render).
- Do NOT open a new PR scope discussion. This is a surgical fix.

## Verification

After deploy, manually trigger a run from the Render dashboard:
Overdue / Late-Fee Reminders & Charges → "Trigger Run" button.

Expected:
1. Run completes in well under the cron timeout (typically <1 minute
   for an empty-overdue day).
2. Render event feed shows "Successful run" (NOT "Failed run").
3. stdout ends with `✅ Overdue reminders script complete, exiting.`
   and exit code 0.
4. No `[redis] connect / [redis] ready` reconnection loops after the
   summary block.
5. If there were any `chargedCount > 0`, the daily digest email lands
   at `amalyb@gmail.com` (SendGrid key was added to this cron's env
   vars earlier today).

No regression test needed — behavior change is process-lifecycle only
and is verified by the Render "Successful run" status after deploy.
