## B1 — `OVERDUE_FEES_CHARGING_ENABLED` undefined

**File:** `server/scripts/sendOverdueReminders.js:590`

**Current behavior:** The identifier `OVERDUE_FEES_CHARGING_ENABLED` is used
inside the `overdue.charge.skip` JSON-log event but is never imported or
declared in this file. Only `applyCharges` is destructured from `lateFees.js`
at line 48, and `lateFees.js` doesn't export the const. Every skip-path log
emit throws `ReferenceError` inside the `try`, gets caught as
`overdue.charge.error`, and bumps `chargesFailed++`.

**Bug:** Current pre-flip dry-run logs show every overdue tx as a charge
failure, which defeats PR-4 dry-run analysis. The digest email's `skipped_*`
buckets are also wrong.

**Fix:** Read from `process.env` directly, matching the existing pattern at
line 213:

```js
flag: process.env.OVERDUE_FEES_CHARGING_ENABLED === 'true' ? 'LIVE' : 'DISABLED'
```

**Regression test:** Add a unit test that exercises the skip path with
`process.env.OVERDUE_FEES_CHARGING_ENABLED` both set and unset, asserting the
structured log serializes without throwing.

## B2 — Orphan `transition/store-shipping-urls`

**Files:**
- `server/webhooks/shippoTracking.js:1346`
- `server/webhooks/shippoTracking.js:1353`
- `server/scripts/resendDeliverySms.js:128`

**Current behavior:** Outbound first-scan and delivered branches call
`sdk.transactions.transition({ transition: 'transition/store-shipping-urls',
params: { protectedData } })` to persist `outbound.firstScanAt`,
`shippingNotification.firstScan.sent`, `shippingNotification.delivered.sent`,
and `lastTrackingStatus.status`.

**Bug:** `transition/store-shipping-urls` does NOT exist in any version of
`process.edn`. The call 400s with `unknown-transition`, gets caught at
shippoTracking.js:1360–1363, and is logged as "Failed to update transaction
protectedData". No protectedData ever lands. Cascade:
1. `outbound.firstScanAt` never set → `hasOutboundScan()` returns false for
   every shipped tx → `sendAutoCancelUnshipped` will cancel shipped bookings
   once `AUTO_CANCEL_DRY_RUN=0`.
2. `shippingNotification.delivered.sent` never set → webhook retries on
   delivery re-send SMS to borrower.
3. `sendShippingReminders.isOutboundScanned()` always returns false → 24h /
   EOD / 48h reminders all still fire for shipped items.

**Fix:** Replace each `sdk.transactions.transition({transition: 'transition/
store-shipping-urls', ...})` call with:

```js
await upsertProtectedData(txId, protectedDataUpdate, { source: 'webhook' });
```

**Reference pattern:** `server/webhooks/shippoTracking.js:901` (return
first-scan) and `:1102` (return delivered) already use `upsertProtectedData`
correctly.

**Important behavioral note:** `upsertProtectedData` does a deep merge, not a
replace. Each call site must pass the intended SHAPE. For outbound first-scan,
pass:

```js
{
  outbound: { firstScanAt: <iso-timestamp> },
  shippingNotification: { firstScan: { sent: true, sentAt: <iso> } },
  lastTrackingStatus: { status: <status-string>, updatedAt: <iso> },
}
```

— not a flat `{ firstScanAt: ... }` at the top level. Read the existing
working calls in the same file and mirror their shape conventions.

**Regression tests:**
1. Integration test for the outbound first-scan webhook handler: simulate a
   Shippo "TRANSIT" event; after handling, assert protectedData has
   `outbound.firstScanAt` set, `shippingNotification.firstScan.sent === true`,
   `lastTrackingStatus.status === 'TRANSIT'`.
2. Webhook-retry idempotency test: fire the same TRANSIT event twice;
   assert the SMS (4-SMS) sends exactly once (the in-memory dedupe AND
   the now-persistent protectedData should both gate it).
3. Delivered branch: simulate a "DELIVERED" event; assert
   `shippingNotification.delivered.sent === true` persists; fire the event
   twice and assert the delivery SMS (6-SMS) sends exactly once.

## B3 — Double-cancel path

**File:** `server/scripts/sendShippingReminders.js:237-241`

**Current behavior:** When `hoursAfterShipBy >= 48 && < 72`, the script calls
`transition/cancel` (full refund) and sends an SMS to the lender only.

**Bug:** This path overlaps with `sendAutoCancelUnshipped` (which cancels at
end-of-D lender-local). While `AUTO_CANCEL_DRY_RUN=1`, this legacy path is the
only cancel path actually firing in prod. Once `AUTO_CANCEL_DRY_RUN=0`, both
paths race. This path also sends NO SMS to the borrower — borrower is silently
refunded with no notification. And because of B2, `isOutboundScanned()` always
returns false, so even shipped items get cancelled at 48h.

**Fix:** DELETE the 48h auto-cancel block entirely. `sendAutoCancelUnshipped`
owns cancel policy for unshipped bookings now. Also delete any helpers or
constants that become unreferenced, and any scenario tests that exclusively
exercised this path (update tests that incidentally exercised it).

**Do NOT env-gate.** Env-gating defers the decision; deletion makes ownership
explicit.

Add a one-line comment at the top of `sendShippingReminders.js` (near the
module docblock):

```js
// Cancel policy for unshipped bookings lives in sendAutoCancelUnshipped.js.
// This script only sends ship-by reminders (24h, EOD); it does NOT cancel.
```

## H1 — Quiet-hours gate below Redis lock

**File:** `server/scripts/sendReturnReminders.js:621-642`

**Current behavior:** `acquireRedisLock(perTxLockKey, 60*60*24)` runs at line
621. Then at line 639, the code checks `if (!withinSendWindow()) continue`.

**Bug:** If the first cron tick of the day lands during quiet hours (e.g.
7:45 AM PT), the script acquires the 24-hour lock and then skips. Every
subsequent tick that day hits "lock already held" and also skips. The
T-1/TODAY reminder is lost for the whole day. The Pattern-A claim in PR-3b
("15-min poll retries naturally in the send window") is broken.

**Fix:** Move the `withinSendWindow()` check ABOVE `acquireRedisLock`:

```js
if (!withinSendWindow(getNow())) {
  log('[RETURN] SKIP quiet-hours tx=' + txId);
  continue;
}
// ... then acquireRedisLock ...
```

**Regression test:** Simulate two cron ticks via `FORCE_NOW`: first at
7:45 AM PT (should skip, NO lock acquired), second at 9:00 AM PT (should send
SMS; lock now acquired and held for 24h). Assert lock state between ticks.

## H2 — `sendReturnReminders` still has `sdk.transactions.update`

**File:** `server/scripts/sendReturnReminders.js:189`

**Current behavior:** `updateTransactionProtectedData` helper attempts
`txApi.update` → `updateMetadata` → `updateProtectedData` in sequence. The
Integration SDK exposes NONE of these methods for transactions. All writes
throw, are caught at lines 678/702/739, and logged as "Failed to mark ... as
sent."

**Bug:** Third instance of the `sdk.transactions.update` silent-fail pattern
this codebase has seen (after `sendLenderRequestReminders.js` and
`sendShippingReminders.js` in April 2026). Per-day Redis lock already covers
single-send-per-day, but cross-process flags (`tMinus1SentAt`,
`todayReminderSentAt`, `tomorrowReminderSentAt`,
`returnSms.dueTodayLastSentLocalDate`) never persist, and every cron tick
produces noisy error logs.

**Fix:** Replace the entire `updateTransactionProtectedData` helper call with
`upsertProtectedData(txId, { returnSms: { ... } }, { source: 'return-reminders' })`.
Import `upsertProtectedData` from `server/lib/txData.js`. Delete the
`updateTransactionProtectedData` helper if nothing else references it.

**Regression test (extend H1's):** after the SMS fires in the send-window
tick, assert `protectedData.returnSms.tMinus1SentAt` (or whichever flag the
branch writes) is actually set on the tx.
