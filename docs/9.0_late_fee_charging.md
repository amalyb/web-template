# 9.0 — Late Fee Charging Fix (P0)

Owner: backend
Priority: P0 (revenue-critical — fees are currently not being charged in prod)
Related files:
- `server/lib/lateFees.js`
- `server/scripts/sendOverdueReminders.js`
- `ext/transaction-processes/default-booking/process.edn` (live)
- `ext/transaction-processes/default-booking_backup_before_pull/process.edn` (backup, contains transitions)

---

## TL;DR

The `applyCharges()` function in `server/lib/lateFees.js` calls two privileged
transitions that **do not exist** in the live `process.edn`:

- `transition/privileged-apply-late-fees`
- `transition/privileged-apply-late-fees-non-return`

Every overdue-reminder cron pass silently throws "unknown-transition" at the
SDK layer. Borrowers receive "$15/day fee is being charged" SMS but are never
actually billed. This is both a revenue leak and a legal/trust exposure.

This doc is scoped to restoring charging safely. Message copy rewrites live
in `9.1_overdue_copy_refactor.md`.

### PR-2: Version gate + day-6 cap inside applyCharges()

Two guards added at the top of `applyCharges()` in `server/lib/lateFees.js`:

```js
const MIN_PROCESS_VERSION_FOR_LATE_FEES = 4; // bump when PR-1 lands
const MAX_CHARGEABLE_DAYS = 6;

if ((tx.attributes?.processVersion || 0) < MIN_PROCESS_VERSION_FOR_LATE_FEES) {
  return { charged: false, reason: 'processVersion-too-old', processVersion: tx.attributes?.processVersion };
}

if (lateDays > MAX_CHARGEABLE_DAYS) {
  return { charged: false, reason: 'exceeded-max-chargeable-days', lateDays };
}
```

Why inside the function, not at cron level: older-process txs still receive
SMS (handled independently). Charging skips with a structured, logged reason.

Acceptance:
- DRY_RUN against a v1/v2 tx logs `reason: 'processVersion-too-old'`.
- DRY_RUN against a v3+ tx that's 7 days late logs
  `reason: 'exceeded-max-chargeable-days'`.
- Neither case throws.

### PR-4: Structured error logging + Slack alert

In `sendOverdueReminders.js`, wrap every `applyCharges()` call with:

```js
try {
  const result = await applyCharges({ sdkInstance, txId, now });
  logStructured('overdue.charge', { txId, ...result });
} catch (err) {
  const code = err?.originalError?.data?.errors?.[0]?.code;
  logStructured('overdue.charge.error', { txId, code, message: err.message });
  if (code === 'unknown-transition') {
    await notifySlack('#eng-alerts', `🚨 overdue-cron unknown-transition for tx ${txId} — process.edn deployment may have failed`);
  }
}
```

Metrics to track (stdout JSON, parsed by Render log drain):
- `overdue.charge` with `charged`, `lateDays`, `items`, `scenario`
- `overdue.charge.error` with `code`, `message`
- `overdue.skip` with `reason` (`processVersion-too-old`,
  `exceeded-max-chargeable-days`, `no-op`, `not-overdue`)

Acceptance:
- Forced `unknown-transition` (rename transition temporarily in a staging
  test) triggers the Slack alert.
- All paths emit one structured log line per tx.

### PR-5: Feature flag `OVERDUE_FEES_CHARGING_ENABLED` + rollout

- Add env var `OVERDUE_FEES_CHARGING_ENABLED` (default `false`) to
  `server/lib/lateFees.js`:

```js
const OVERDUE_FEES_CHARGING_ENABLED =
  String(process.env.OVERDUE_FEES_CHARGING_ENABLED || 'false').toLowerCase() === 'true';
```

- Wrap the `sdkInstance.transactions.transition(...)` call:

```js
if (!OVERDUE_FEES_CHARGING_ENABLED) {
  return {
    charged: false,
    reason: 'feature-flag-disabled',
    wouldCharge: newLineItems.map(i => ({ code: i.code, cents: i.unitPrice.amount })),
    lateDays,
    scenario,
  };
}
```

- SMS sending is unaffected by this flag.

Rollout sequence (no redeploy for flag flip — Render env var):
1. Ship PRs 1–5 with `OVERDUE_FEES_CHARGING_ENABLED=false` in Render.
2. Monitor 24h: SMS sends normally; `overdue.charge` logs show
   `reason: 'feature-flag-disabled'` with `wouldCharge` amounts.
3. Spot-check 5 txs — confirm amounts match expectations.
4. Flip `OVERDUE_FEES_CHARGING_ENABLED=true` in Render console.
5. Monitor 24h: charges appear in Stripe dashboard, `chargeHistory`
   populates, no `unknown-transition` alerts.
6. If success rate > 95%, mark stable.

Rollback: flip flag back to `false`. No redeploy, no code change.

## Edge cases to handle

### Stripe off-session payment failures
- Card declined, insufficient funds, cancelled card between booking and Day N.
- `applyCharges()` should catch the Stripe error, mark
  `protectedData.return.chargeFailedOnDay = N` with the error code, and skip
  further attempts for that day (idempotency advances anyway so next day tries
  fresh).
- Surface as a structured log + Slack alert for operator follow-up.

### Refund-when-scanned-early
- Borrower charged $15 on Day 1, package scans Day 2.
- Current behavior: fees are locked in via idempotency; no auto-refund.
- Recommendation: keep manual. Add `chargeHistory` entries with
  `scannedAt` timestamp so operators can process refunds from the Console.

### Old-processVersion transactions overdue during rollout
- Version gate (PR-2) skips them safely with logged reason.
- SMS still sends so the borrower isn't silently dropped.
- Surface count in daily log summary.

### Cron runs past the Day-6 soft limit
- PR-2's `MAX_CHARGEABLE_DAYS = 6` guard inside `applyCharges()` is the
  authoritative cap. SMS hard-stop at Day 6 in the cron is the outer layer;
  `applyCharges()` cap is the inner layer. Defense in depth.
