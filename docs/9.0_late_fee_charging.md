# 9.0 — Late Fee Charging Fix (P0)

Owner: backend
Priority: P0 (revenue-critical — fees are currently not being charged in prod)
Related files:
- `server/lib/lateFees.js`
- `server/scripts/sendOverdueReminders.js`
- `ext/transaction-processes/default-booking/process.edn` (live)
- `ext/transaction-processes/default-booking_backup_before_pull/process.edn` (backup, contains transitions)

---

## TL;DR

The `applyCharges()` function in `server/lib/lateFees.js` calls two privileged
transitions that **do not exist** in the live `process.edn`:

- `transition/privileged-apply-late-fees`
- `transition/privileged-apply-late-fees-non-return`

Every overdue-reminder cron pass silently throws "unknown-transition" at the
SDK layer. Borrowers receive "$15/day fee is being charged" SMS but are never
actually billed. This is both a revenue leak and a legal/trust exposure.

This doc is scoped to restoring charging safely. Message copy rewrites live
in `9.1_overdue_copy_refactor.md`.

## Implementation Plan — 5 atomic PRs

**Critical ordering note:** The feature flag + guards (PR-1) MUST ship BEFORE
the transitions (PR-2). If transitions ship first, the next cron tick calls
`applyCharges()` with no guard and immediately bills every overdue tx in
prod. The flag ships defaulted to `false` so even after the transitions exist,
nothing charges until we explicitly flip the flag in Render console.

---

### PR-1: Feature flag + version gate + day-6 cap + override hook (zero-risk, ships first)

All defensive code added to `server/lib/lateFees.js` BEFORE any process.edn
change. Safe to deploy anytime because `applyCharges()` will still fail on
the missing transition — but now fails with a clean structured return, not
a thrown error, because the flag short-circuits the SDK call.

Four changes in `lateFees.js`:

1. Top-of-file constants:
```js
const LATE_FEE_CENTS = process.env.LATE_FEE_CENTS_OVERRIDE
  ? parseInt(process.env.LATE_FEE_CENTS_OVERRIDE, 10)
  : 1500;

const OVERDUE_FEES_CHARGING_ENABLED =
  String(process.env.OVERDUE_FEES_CHARGING_ENABLED || 'false').toLowerCase() === 'true';

// Confirm the correct number by reading from Sharetribe Console after PR-2's
// alias flip. Current live process is v3; the push in PR-2 will bump it.
// Update this constant in the same PR that flips the alias.
const MIN_PROCESS_VERSION_FOR_LATE_FEES = 3;

const MAX_CHARGEABLE_DAYS = 6;
```

2. Process-version gate at top of `applyCharges()`:
```js
if ((tx.attributes?.processVersion || 0) < MIN_PROCESS_VERSION_FOR_LATE_FEES) {
  return { charged: false, reason: 'processVersion-too-old', processVersion: tx.attributes?.processVersion };
}
```

3. Day-6 cap after `lateDays` is computed:
```js
if (lateDays > MAX_CHARGEABLE_DAYS) {
  return { charged: false, reason: 'exceeded-max-chargeable-days', lateDays };
}
```

4. Feature-flag short-circuit immediately BEFORE
   `sdkInstance.transactions.transition(...)`:
```js
if (!OVERDUE_FEES_CHARGING_ENABLED) {
  return {
    charged: false,
    reason: 'feature-flag-disabled',
    wouldCharge: newLineItems.map(i => ({ code: i.code, cents: i.unitPrice.amount })),
    lateDays,
    scenario,
  };
}
```

SMS sending is unaffected by any of these — SMS lives in
`sendOverdueReminders.js` and is decoupled from `applyCharges()`.

Acceptance:
- Deployed with `OVERDUE_FEES_CHARGING_ENABLED=false` in Render env.
- Next cron pass logs `reason: 'feature-flag-disabled'` with
  `wouldCharge` line items for overdue txs.
- No `unknown-transition` errors — the SDK call is short-circuited before it
  would run.
- Old-version txs log `reason: 'processVersion-too-old'`.

Rollback: revert the PR. No schema or process.edn state to unwind.

### PR-4: Structured error logging (no Slack)

The codebase currently has no Slack webhook integration. We rely on
structured stdout JSON parsed by Render's log drain. Slack/email alerting is
a separate follow-up not scoped here.

In `sendOverdueReminders.js`, wrap every `applyCharges()` call with:

```js
try {
  const result = await applyCharges({ sdkInstance, txId, now });
  console.log(JSON.stringify({ event: 'overdue.charge', txId, ...result }));
} catch (err) {
  const code = err?.originalError?.data?.errors?.[0]?.code;
  console.error(JSON.stringify({
    event: 'overdue.charge.error',
    txId,
    code,
    message: err.message,
  }));
}
```

Metrics to track via Render log drain:
- `overdue.charge` with `charged`, `lateDays`, `items`, `scenario`.
- `overdue.charge.error` with `code`, `message`. An `unknown-transition`
  here means the process.edn push (PR-2) regressed — treat as a page.
- Skip reasons (`processVersion-too-old`, `exceeded-max-chargeable-days`,
  `no-op`, `not-overdue`, `feature-flag-disabled`) ride on the
  `overdue.charge` event via the `reason` field.

Acceptance:
- Forced `unknown-transition` in staging (temporarily rename the transition)
  produces a parseable `overdue.charge.error` JSON line.
- Every `applyCharges()` call emits exactly one structured log line.

Follow-up (out of scope): wire these log events to a Slack or email
notifier. Defer until we see real alerting need.

## Edge cases to handle

### Stripe off-session payment failures
- Card declined, insufficient funds, cancelled card between booking and Day N.
- `applyCharges()` should catch the Stripe error, mark
  `protectedData.return.chargeFailedOnDay = N` with the error code, and skip
  further attempts for that day (idempotency advances anyway so next day tries
  fresh).
- Surface as a structured log + Slack alert for operator follow-up.

### Refund-when-scanned-early
- Borrower charged $15 on Day 1, package scans Day 2.
- Current behavior: fees are locked in via idempotency; no auto-refund.
- Recommendation: keep manual. Add `chargeHistory` entries with
  `scannedAt` timestamp so operators can process refunds from the Console.

### Old-processVersion transactions overdue during rollout
- Version gate (PR-2) skips them safely with logged reason.
- SMS still sends so the borrower isn't silently dropped.
- Surface count in daily log summary.

### Cron runs past the Day-6 soft limit
- PR-2's `MAX_CHARGEABLE_DAYS = 6` guard inside `applyCharges()` is the
  authoritative cap. SMS hard-stop at Day 6 in the cron is the outer layer;
  `applyCharges()` cap is the inner layer. Defense in depth.
