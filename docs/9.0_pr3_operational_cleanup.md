# PR-3a — Charging-path correctness

## 3a.1 — Migrate `lastNotifiedDay` persistence off missing transitions, onto Redis

### Current state (the bug)

`server/scripts/sendOverdueReminders.js:698-702` calls a privileged
transition to persist `lastNotifiedDay` to Flex after each SMS send:

```js
const overdueTransition =
  currentState === 'delivered'
    ? 'transition/privileged-set-overdue-notified-delivered'
    : 'transition/privileged-set-overdue-notified';
```

**Neither transition exists in `default-booking/process.edn`** (not in v3, not
in v4, not in the `_backup_before_pull/` copy). Every SMS send silently
fails this transition call; the error is caught and logged, SMS still
dispatches, but `protectedData.return.overdue.lastNotifiedDay` never
persists to Flex.

Dedupe today relies entirely on the in-memory `runNotificationGuard` Map
inside `sendOverdueReminders.js`. A cron restart mid-run (deploy, OOM,
Render restart) wipes that Map, and the next tick can re-send the same
day's SMS to borrowers we've already notified today.

### Why Redis, not add-the-transitions

Two valid fix paths:

| | Option A: add to `process.edn` | Option B: Redis idempotency |
|-|-|-|
| Flex-side visibility | Yes (operator can see `lastNotifiedDay` in Console) | No — Redis only |
| Deploy cost | Another flex-cli push + alias flip (v5) | None |
| Matches existing pattern | No — would be the first notification-dedupe transition in the schema | **Yes** — identical to `shippingReminder:*` + `lenderReminder:*` shipped April 2026 |
| Survives Redis flush | Yes | No (7-day TTL; 6-day overdue window fits with margin) |
| Survives Integration SDK quirks | N/A | **Yes** — Integration SDK doesn't expose `transactions.update`, which is exactly why shipping/lender reminders migrated to Redis |

**Decision: Option B.** Matches the pattern every other SMS dedupe in this
codebase already uses, avoids another process deploy, and trades
theoretical Flex visibility for zero incremental infra risk. The 7-day
Redis TTL comfortably outlasts the 6-day overdue window.

### Implementation

Constants (matching `sendShippingReminders.js` exactly):

```js
const SENT_TTL_SEC     = 7 * 24 * 60 * 60; // 7 days
const INFLIGHT_TTL_SEC = 10 * 60;           // 10 min
```

Keys (pattern mirrors `shippingReminder:*` in `sendShippingReminders.js`):

- `overdueNotified:{txId}:{daysLate}:sent` — set after successful SMS send, 7-day TTL
- `overdueNotified:{txId}:{daysLate}:inFlight` — set before SMS send, 10-min TTL (auto-clears on process crash so the next tick can retry)

Flow (replaces current `runNotificationGuard.set(...)` + failed transition call):

```js
const sentKey = `overdueNotified:${txId}:${daysLate}:sent`;
const inFlightKey = `overdueNotified:${txId}:${daysLate}:inFlight`;

if (await redis.get(sentKey)) {
  console.log(`[OVERDUE][ALREADY-SENT] tx=${txId} day=${daysLate}`);
  continue;
}
if (await redis.get(inFlightKey)) {
  console.log(`[OVERDUE][IN-FLIGHT] tx=${txId} day=${daysLate}`);
  continue;
}

// Use SET NX EX for atomic set-if-absent on the sent mark (closes
// the double-send race if two cron processes overlap on restart).
// Pattern matches sendReturnReminders.js:243.
if (DRY_RUN) {
  console.log(`[OVERDUE][DRY-RUN] Would SET ${sentKey}`);
  continue; // DRY_RUN skips Redis writes too
}

await redis.set(inFlightKey, '1', 'EX', INFLIGHT_TTL_SEC);

try {
  const smsResult = await sendSMS(borrowerPhone, message, { ... });
  if (smsResult?.success) {
    // Atomic: only set if not already set (NX). Prevents double-mark
    // if two processes raced past the get() check above.
    await redis.set(sentKey, String(daysLate), 'NX', 'EX', SENT_TTL_SEC);
    await redis.del(inFlightKey);
  } else {
    await redis.del(inFlightKey); // let next tick retry
  }
} catch (err) {
  await redis.del(inFlightKey);
  throw err;
}
```

The existing `runNotificationGuard` in-process Map is kept as a
belt-and-suspenders second layer — prevents duplicate sends **within** a
single cron run, even if Redis round-trips haven't landed yet.

The dead transition-call block at lines 697–716 is deleted. The TODO comment
added in PR-2 is deleted. (This resolves the "Fix missing
privileged-set-overdue-notified transitions" task — we chose Redis over
adding the transitions to process.edn, per §3a.1 decision rationale.)

### Also required: `scripts/diagnose-overdue.js` migration

`diagnose-overdue.js:165` currently reads `returnData.overdue.lastNotifiedDay`
from Flex protectedData. After 3a.1, that field is no longer authoritative
(it was never being written). Migrate the diagnostic to read from Redis:

```js
const redis = await getRedis();
for (let d = 1; d <= 6; d++) {
  const sent = await redis.get(`overdueNotified:${txId}:${d}:sent`);
  if (sent) console.log(`  Day ${d}: notified (Redis)`);
}
```

If Redis is unavailable, log a warning and note the field is stale.

### Risks / edge cases

- **Redis down.** Falls through to `runNotificationGuard` only (same as
  shipping reminders today). Acceptable — we don't block SMS on Redis.
- **TTL expiry within window.** 7-day TTL vs 6-day window = 24h margin.
  Not a real risk, but if we ever extend the overdue window past 7 days
  we need to revisit the TTL.
- **Deploy-time consistency with other reminders.** Same exact key
  pattern (`{feature}:{txId}:{phase}:{sent|inFlight}`) — operator mental
  model stays consistent.

## 3a.2 — Unified daily charging with 24h scan-lag check

### Policy (confirmed April 15, 2026)

Late fees are **$15/day, charged daily, max 5 charges = $75 cap**.
There is no lump-sum "catch-up" charge. Every charge is exactly $15,
billed one day at a time. When a USPS scan arrives showing the item
has been dropped off, daily charges stop — no further fees regardless
of how many days of lateness remain.

The Scenario A / Scenario B distinction in the code (`delivered` +
scan vs `accepted` + no scan) matters for **state routing** (which
transition to call) but NOT for charge amount. Both scenarios charge
$15/day, quantity 1. The only difference is: once a scan appears,
charging stops.

### The scan-lag rule

USPS scans can take up to 24 hours to post after physical drop-off.
To avoid charging a borrower who already returned the item but whose
scan hasn't posted yet:

**Rule: on day N's cron run, charge $15 for day N-1 only if no scan
has appeared by cron run time. Day boundaries use PT (America/Los_Angeles),
not UTC.**

Walk-through (item due Monday):

| Cron day | SMS | Charge | Check |
|-|-|-|-|
| Tue (day 1 late) | Day-1 SMS ("may apply") | No charge | No prior late day to charge for |
| Wed (day 2 late) | Day-2 SMS ("applies") | $15 for Tue (charge #1) | Any scan in system? If yes → stop |
| Thu (day 3 late) | Day-3 SMS | $15 for Wed (charge #2) | Any scan? If yes → stop |
| Fri (day 4 late) | Day-4 SMS | $15 for Thu (charge #3) | Any scan? If yes → stop |
| Sat (day 5 late) | Day-5 SMS | $15 for Fri (charge #4) | Any scan? If yes → stop |
| Sun (day 6 late) | Day-6 SMS (replacement warning) | $15 for Sat (charge #5, final) | Any scan? If yes → stop |
| Day 7+ | Hard stop — no SMS, no charge | — | — |

Key behaviors:
- **Day 1 SMS goes out before any charge.** Copy says "may apply" not
  "now applies" — see 9.1 doc amendment.
- **Day 6 cron charges $15 for day 5's lateness AND sends the replacement
  SMS.** The SMS doesn't mention the $15 (it's about replacement), but
  the charge still lands. This is the 5th and final $15 = $75 total.
- **Scan arrival at any point stops everything.** If scan posts on
  Thursday afternoon (during day 3 late), Friday's cron sees the scan
  and sends no SMS, charges nothing. Net charges = $30 (for Tue + Wed).
- **Two idempotency systems remain separate:** SMS dedupe → Redis
  (`overdueNotified:*`). Charge dedupe → Flex protectedData
  (`lastLateFeeDayCharged`). They do not cross-reference.

### Current state (the bug)

`server/lib/lateFees.js:244-258` — Scenario A branch pins
`effectiveDate = scanDate`. Since `scanDate` never changes,
`lastLateFeeDayCharged === ymd(scanDate)` blocks every tick after the
first. Result: Scenario A charges **$15 once** regardless of lateness.

### Fix approach

Remove the Scenario A special path entirely. Both scenarios use the
same charging logic: `effectiveDate = now` (today's date), `quantity: 1`,
`$15` per cron tick. The "did they return?" check becomes:

```js
// Before charging, check: has a scan appeared for this tx?
if (hasScan) {
  console.log(`[lateFees] SKIP tx=${txId} reason=scan-detected scanDate=${ymd(scanDate)}`);
  return { charged: false, reason: 'scan-detected', scenario: 'returned' };
}
```

This replaces the entire Scenario A branch (`lateFees.js:246-260` in
pre-3a code) and the `lateFeeScenarioACharged` flag /
`priorScenarioBDaysCharged` counter from the previous version of this
doc. Those are no longer needed because there is no lump-sum catch-up
— only daily ticks that stop when a scan appears. **Delete the dead
Scenario A code path entirely** — do not leave it commented out.

### Implementation (lateFees.js changes)

In `applyCharges()`, replace the Scenario A / B branching with:

```js
// Unified daily charging — both scenarios use the same logic.
// The only question is: has a scan appeared? If yes, stop charging.
if (hasScan) {
  console.log(`[lateFees] SKIP tx=${txId} reason=scan-detected ` +
    `scanDate=${ymd(scanDate)} returnDueAt=${ymd(returnDueAt)}`);
  return { charged: false, reason: 'scan-detected' };
}

// Explicit scan-lag grace: day 1 cron never charges. The scan-lag rule
// says "charge for day N-1 on day N's cron," and there is no day 0 to
// charge for. This guard codifies that as a code-level gate rather than
// relying solely on cron timing — if the cron ever fires early or a
// manual invocation passes daysLate=1, this prevents a false charge.
if (daysLate <= 1) {
  console.log(`[lateFees] SKIP tx=${txId} reason=scan-lag-grace daysLate=${daysLate}`);
  return { charged: false, reason: 'scan-lag-grace', daysLate };
}

// No scan yet — charge $15 for yesterday's lateness.
const effectiveDate = new Date(); // today, advances daily
const effectiveYmd = ymd(effectiveDate);

if (lastLateFeeDayCharged === effectiveYmd) {
  return { charged: false, reason: 'already-charged-today' };
}

// Cap check: max 5 charges total = $75.
const priorChargeCount = (returnData.chargeHistory || [])
  .filter(e => e.items?.some(i => i.code === 'late-fee'))
  .length;

if (priorChargeCount >= 5) {
  console.log(`[lateFees] SKIP tx=${txId} reason=max-charges-reached count=${priorChargeCount}`);
  return { charged: false, reason: 'max-charges-reached', chargeCount: priorChargeCount };
}

// Build the line item — always quantity: 1, always $15.
// Scenario value for chargeHistory entries: 'daily-overdue'.
// Old entries created before this change used 'non-return' — the cap
// counter filters by `code === 'late-fee'`, not scenario, so old
// entries are backward-compatible and still count toward the $75 cap.
newLineItems.push({
  code: 'late-fee',
  unitPrice: { amount: LATE_FEE_CENTS, currency: 'USD' },
  quantity: 1,
  percentage: 0,
  includeFor: ['customer'],
});
const chargeScenario = 'daily-overdue';
console.log(`[lateFees] Charging $${LATE_FEE_CENTS / 100} for tx=${txId} ` +
  `day=${daysLate} effective=${effectiveYmd} chargeCount=${priorChargeCount + 1}/5`);
```

Under the unified model, `applyCharges()` only ever calls
`transition/privileged-apply-late-fees-non-return` (from
`:state/accepted`, process.edn:148). Any `:state/delivered` tx is
intercepted upstream by either the `hasScan` check (`scan-detected`
skip) or the PR-2 `delivered-without-scan` policy-skip before reaching
the transition call. There is no branching on transition name — the
code uses one transition only.

**Vestigial transition note:** `transition/privileged-apply-late-fees`
(from `:state/delivered`, process.edn:128) was added in process.edn v4
but is now effectively unused under the unified model — no
`:state/delivered` tx ever reaches the transition call. **Document this
in a code comment; do not remove from process.edn** (removing would
require a v5 push + alias flip for zero behavioral benefit).

### Why this is simpler than the previous spec

The previous version of this doc proposed:
- `lateFeeScenarioACharged` flag — **removed** (no lump-sum to gate)
- `priorScenarioBDaysCharged` counter — **removed** (no B→A handoff math)
- `quantity: lateDays` one-shot — **removed** (always quantity: 1)
- B→A handoff reconciliation — **removed** (scan stops charging, period)

What remains: one unified path, `hasScan` check at the top, daily $15,
cap at 5, done.

## 3a.3 — Move overdue cron from 09:00 UTC to 17:00 UTC

### Why

09:00 UTC = **1 AM PT / 4 AM ET**. Borrowers receive late-fee SMS at
1 AM. Unacceptable.

17:00 UTC year-round gives:
- Winter (PST): **9 AM PT** / 12 PM ET
- Summer (PDT): **10 AM PT** / 1 PM ET

Both are well inside business hours. Also improves the scan-lag buffer:
at 17:00 UTC, any scan from the previous calendar day (PT) has had
9–10 hours to post, vs 1 hour at the current 09:00 UTC.

### Implementation

In `sendOverdueReminders.js`, daemon scheduling block (~line 810):

```js
// BEFORE:
// next9AM.setUTCHours(9, 0, 0, 0);
// AFTER:
next9AM.setUTCHours(17, 0, 0, 0);
```

Update the surrounding variable names and log messages (`next9AM` →
`nextRunTime`, comments from "9 AM UTC" to "17:00 UTC / ~10 AM PT").

Update `render.yaml` comment (line 44 area) to reflect the new time.

## 3a.4 — Charging cap: 5 charges / $75 max

### Policy

Max 5 daily $15 charges = **$75 total late fees**. After 5 charges,
the cron stops charging even if the item is still out. Day 6 is the
replacement-warning SMS; no 6th $15 charge accompanies it (the 5th
charge was already collected by day 6's cron for day 5's lateness).

Day 7+: hard stop. No SMS, no charge, no further cron processing.

### Implementation

Replace `MAX_CHARGEABLE_DAYS` logic in `lateFees.js` with a
charge-count cap:

```js
const MAX_LATE_FEE_CHARGES = 5;

const priorChargeCount = (returnData.chargeHistory || [])
  .filter(e => e.items?.some(i => i.code === 'late-fee'))
  .length;

if (priorChargeCount >= MAX_LATE_FEE_CHARGES) {
  return { charged: false, reason: 'max-charges-reached', chargeCount: priorChargeCount };
}
```

This is count-based (how many times we've charged), not day-based
(how many days late). Count is the correct metric because missed cron
runs could skip a day, and we shouldn't penalize the borrower for
our infrastructure gaps. If cron missed day 3 and charged days 1, 2,
4, 5 — that's 4 charges ($60), and they get one more before the cap.

# PR-3b — Operational emails + SMS quiet hours

## 3b.1 — Day-6 fire-and-forget email alert

### Why

The day-6 SMS reads: *"🚨 Your return is now 6 days late... the full
replacement value of the item may be charged."* That's a strong warning
but not a definite commitment (see 9.1 doc — "may" not "will"). Auto-replacement is intentionally disabled
(`AUTO_REPLACEMENT_ENABLED = false`), so the actual replacement charge
is manual operator action. The operator needs a reliable cue that a
day-6 SMS fired so they can decide whether to trigger the manual
replacement before the promise ages poorly.

### Implementation

Path: `server/scripts/sendOverdueReminders.js`, next to the existing
`[OVERDUE][DAY6_SENT]` console.log (around line 663 in current code).

```js
const { sendTransactionalEmail } = require('../email/emailClient');

// ... inside the daysLate === 6 branch after SMS send succeeds ...

// Derive borrower display name from included customer resource
const customerName =
  customer?.attributes?.profile?.displayName ||
  customer?.attributes?.profile?.firstName ||
  'unknown';

if (daysLate === 6 && smsResult?.sent) {
  console.log('[OVERDUE][DAY6_SENT]', { txId: tx?.id?.uuid || tx?.id, daysLate, tag });

  // Fire-and-forget operator alert — .catch() prevents unhandledRejection
  // from crashing the cron if SendGrid is down. Do NOT await.
  const emailBody = [
    `Day-6 SMS just fired to borrower — replacement charge commitment is now in effect.`,
    ``,
    `Tx: ${tx?.id?.uuid || 'unknown'}`,
    `Listing: ${listing?.attributes?.title || 'unknown'} (${listing?.id?.uuid || 'unknown'})`,
    `Borrower: ${customerName} (${borrowerPhone})`,
    `Return due: ${ymd(returnDate)}`,
    `Days late: ${daysLate}`,
    ``,
    `Action: decide whether to trigger manual replacement charge.`,
    `Console: https://console.sharetribe.com/o/sherbrt/transactions/${tx?.id?.uuid}`,
  ].join('\n');

  if (!DRY_RUN) {
    sendTransactionalEmail({
      to: process.env.OPS_ALERT_EMAIL || 'amalyb@gmail.com',
      subject: `[Sherbrt] Day-6 overdue — manual replacement decision needed (tx ${tx?.id?.uuid?.slice(0, 8) || '??'})`,
      text: emailBody,
    }).catch(err => {
      console.error('[OVERDUE][DAY6_EMAIL_FAILED]', { txId: tx?.id?.uuid, error: err.message });
    });
  } else {
    console.log('[OVERDUE][DAY6_EMAIL_DRY_RUN]', { txId: tx?.id?.uuid });
  }
}
```

Key design notes:
- **Fire-and-forget with `.catch()`** — `sendTransactionalEmail` throws
  on SendGrid failure (emailClient.js:63). Without `.catch()`, the
  un-awaited promise becomes `unhandledRejection` → Node crashes the
  cron mid-run. The `.catch()` logs and swallows; cron continues.
- **DRY_RUN suppresses email sends** — mirrors the SMS wrapper pattern
  at sendOverdueReminders.js:79-91. Prevents noisy test emails.
- **Dedupe via the Redis `overdueNotified:{txId}:6:sent` key from 3a.1.**
  Since the email only fires inside the post-SMS success branch, the
  Redis-gated SMS already guarantees single-send per tx per day. No
  separate email idempotency key needed.
- **Recipient via env var** (`OPS_ALERT_EMAIL`) with hardcoded fallback
  so local dev + CI don't need the env set.

### Test plan

- Unit: mock `sendTransactionalEmail`, assert it's called exactly once
  per day-6 SMS with expected `to`, `subject`, `text` containing txId.
- Integration: run the cron with a day-6 fixture tx, assert the email
  function is invoked and the cron returns cleanly even when the mock throws.
- DRY_RUN: assert email is NOT sent, log line emitted instead.
- Prod verification: first real day-6 SMS after deploy should produce
  a matching email in amalyb@gmail.com inbox.

## 3b.2 — Daily late-fee digest email

### Why

"One email per day listing every tx that was charged that day" gives
amalia end-of-run visibility into total charge volume without flooding
her inbox on high-volume days.

### Implementation approach

**Decision: in-cron digest (Option A).** The overdue cron already
iterates every tx that could be charged today. Collecting events into
an array during the run and emailing at the end is ~40 lines and zero
new infra. The digest is accumulated in memory and sent AFTER the
for-loop finishes, BEFORE process exit. Won't survive a mid-run crash
— acceptable since logs exist, but this is not a retry target.

### Code sketch

At the top of `sendOverdueReminders()`:

```js
const chargeDigest = []; // collect charge events for end-of-run email
```

At the END of each tx iteration (after the SMS block, where `listing`,
`customer`, and `customerName` are all in scope), bucket by charge outcome.
Model B: `day6_hard_stop` bucket dropped — the day-6 alert email (§3b.1)
is the escalation signal; the digest is a money summary only.

```js
// customerName derived earlier in the iteration (see §3b.1 fix):
// const customerName = customer?.attributes?.profile?.displayName
//   || customer?.attributes?.profile?.firstName || 'unknown';

if (chargeResult.charged === true) {
  chargeDigest.push({
    bucket: 'charged',
    txId: tx?.id?.uuid,
    listingTitle: listing?.attributes?.title || 'unknown',
    borrowerName: customerName,
    daysLate: chargeResult.lateDays,
    totalCents: LATE_FEE_CENTS, // always $15 per charge
  });
} else if (chargeResult.charged === false && chargeResult.reason) {
  chargeDigest.push({
    bucket: `skipped_${chargeResult.reason}`,
    txId: tx?.id?.uuid,
    daysLate: chargeResult.lateDays || daysLate,
    totalCents: 0,
  });
}
```

At the end of the loop:

```js
if (chargeDigest.length > 0 && !DRY_RUN) {
  await sendLateFeeDigest(chargeDigest);
}
```

New helper:

```js
async function sendLateFeeDigest(events) {
  const ymdToday = ymd(new Date());
  const charged = events.filter(e => e.bucket === 'charged');
  const skipped = events.filter(e => e.bucket.startsWith('skipped_'));
  const totalCents = charged.reduce((s, e) => s + e.totalCents, 0);
  const totalDollars = (totalCents / 100).toFixed(2);

  const lines = [
    `Sherbrt late-fee digest for ${ymdToday}`,
    ``,
    `Charged: $${totalDollars} across ${charged.length} tx(s)`,
    ...charged.map(e =>
      `  tx ${e.txId?.slice(0, 8)} — "${e.listingTitle}" (${e.borrowerName}) — day ${e.daysLate} — $${(e.totalCents / 100).toFixed(2)}`
    ),
    ``,
    `Skipped: ${skipped.length}`,
    ``,
    `Flag: ${process.env.OVERDUE_FEES_CHARGING_ENABLED === 'true' ? 'LIVE' : 'DISABLED (dry-run)'}`,
  ];

  try {
    await sendTransactionalEmail({
      to: process.env.OPS_ALERT_EMAIL || 'amalyb@gmail.com',
      subject: `[Sherbrt] Late-fee digest ${ymdToday} — $${totalDollars} (${charged.length} charged)`,
      text: lines.join('\n'),
    });
    console.log('[OVERDUE][DIGEST_SENT]', { ymd: ymdToday, charged: charged.length, totalCents });
  } catch (err) {
    console.error('[OVERDUE][DIGEST_FAILED]', { error: err.message });
  }
}
```

### Risks / edge cases

- **Empty day.** If `chargeDigest.length === 0`, no email. Silence is fine.
- **DRY_RUN mode.** Digest is suppressed (`!DRY_RUN` gate). Dry-run
  charge events are visible in Render logs only.
- **Partial failures.** If one tx's Stripe call fails mid-run, other txs
  continue and the digest still sends with the successful charges. Failed
  ones show up in `[OVERDUE][CHARGE_FAILED]` log lines for manual review.
- **Day-6 overlap.** A tx hitting day-6 fires the 5th $15 charge AND
  the day-6 SMS + operator alert email. In the digest, day-6 txs appear
  as `charged` (money summary). The day-6 alert email (§3b.1) is the
  separate escalation signal — no `day6_hard_stop` bucket needed (Model B).

## 3b.3 — Quiet-hours gate: 8 AM – 11 PM PT

### Why

All four SMS-sending daemons (overdue, return, shipping, lender-request)
can currently fire SMS at any hour. The overdue cron's 09:00 UTC = 1 AM
PT is the worst offender (fixed in 3a.3 by moving to 17:00 UTC), but the
15-minute polling daemons can also fire late-night SMS if their trigger
conditions cross during off-hours.

### Implementation

New helper in `server/util/time.js`:

```js
/**
 * Returns true if `now` falls within the SMS send window: 8 AM – 11 PM PT.
 * Used by all SMS-sending daemons to defer messages outside business hours.
 *
 * Defaults to getNow() so FORCE_NOW env var is respected in scenario tests
 * without requiring callers to pass the time explicitly.
 */
function getNow() {
  if (process.env.FORCE_NOW) return new Date(process.env.FORCE_NOW);
  return new Date();
}

function withinSendWindow(now = getNow()) {
  const ptHour = parseInt(
    now.toLocaleString('en-US', {
      timeZone: 'America/Los_Angeles',
      hour: '2-digit',
      hour12: false,
    }),
    10
  );
  return ptHour >= 8 && ptHour < 23; // 8:00 AM – 10:59 PM PT
}

module.exports = { ..., getNow, withinSendWindow };
```

### Application by daemon type

**Pattern A — Wide-window daemons (auto-defer via natural retry):**

`sendReturnReminders.js`, `sendShippingReminders.js`, `sendShipByReminders.js`.
These trigger on conditions with 12–24h windows. Gate at every SMS send:

```js
const { withinSendWindow } = require('../util/time');

// Before each sendSMS() call:
if (!withinSendWindow()) {
  console.log(`[${scriptName}][QUIET-HOURS] tx=${txId} — deferred to next poll`);
  continue;
}
```

The 15-min polling loop naturally retries. First poll inside the 8 AM
window picks up the deferred message. No queuing infrastructure needed.

**Pattern B — Narrow-window daemon (delay to 8 AM, not skip):**

`sendLenderRequestReminders.js`. The [60, 80) minute trigger window is
too narrow for a naive skip — by 8 AM the request is hours past 80 min
and a simple `withinSendWindow()` + original upper bound would filter it
out permanently. Solution: widen the upper bound unconditionally so
overnight-deferred txs remain visible the next morning, AND add a
send-time gate so SMS never fires during quiet hours.

```js
const { withinSendWindow, getNow } = require('../util/time');

// Wide upper bound: 13 hours covers any quiet-hours gap (11 PM → 8 AM
// = 9h, plus margin). Txs older than 13h are genuinely stale.
const UPPER_BOUND_MIN = 13 * 60;

// ... existing filter becomes: ageMinutes >= 60 && ageMinutes < UPPER_BOUND_MIN

// Per-tx loop, before sendSMS():
if (!withinSendWindow(getNow())) {
  console.log(`[lender-reminder][QUIET-HOURS] tx=${txId} age=${ageMinutes}m — deferred`);
  continue;
}
```

Walk-through (request placed at 10:30 PM PT):

| Time (PT) | tx age | `withinSendWindow()` | Filter (`>= 60 && < 780`) | Result |
|-|-|-|-|-|
| 11:30 PM | 60 min | false | eligible | **Deferred** (quiet-hours gate) |
| 11:45 PM | 75 min | false | eligible | Deferred |
| … | … | false | … | Deferred |
| 8:00 AM | 570 min | true | eligible (570 < 780) | **SMS fires** |
| 8:15 AM | 585 min | true | eligible | Redis `sent` mark → skip |

Key behaviors:
- During business hours (8 AM – 11 PM PT), the filter catches txs in
  the [60, 780) min window normally. Most fire around 60–80 min as before.
- During quiet hours, the send-time gate defers. The wide upper bound
  ensures the tx is still eligible when the window reopens at 8 AM.
- The Redis `already-sent` idempotency key prevents double-firing for
  requests that were already reminded during business hours.
- A request placed at 10:30 PM gets its reminder at ~8:00 AM — no
  longer a "60-minute follow-up" but still valuable (lender hasn't acted
  for 9+ hours → definitely worth a nudge).

**Pattern C — Daily cron (already handled):**

`sendOverdueReminders.js`. Moved to 17:00 UTC in §3a.3. Always lands
inside the 8 AM – 11 PM window. No gate needed.

### Edge case: event-driven SMS

Some SMS sends are triggered by transaction events (e.g., booking
acceptance) rather than cron polls. These fire immediately when the
event occurs and have no retry loop. Adding quiet-hours support for
event-driven SMS would require a delay queue (Redis-backed or similar).
**Out of scope for PR-3b** — event-driven sends are infrequent at
current volume and the events themselves (booking accepted, payment
received) are user-initiated actions that imply the user is awake.
